# Memory Management

Metaphor's host device uses a caching allocator backed by a buddy system to reduce allocation overhead. Tensor operations constantly create and discard intermediates, and going to the operating system for each allocation would be prohibitively expensive. The cache absorbs this churn by returning freed memory to an internal pool organized by power-of-two block sizes. Allocation finds the smallest sufficient block, splitting larger blocks as needed. Freeing attempts to merge adjacent blocks to reduce fragmentation. A hash map keyed by pointer provides constant-time free without external context. The entire pool is mapped as a large contiguous region upfront with lazy commit, functioning like an arena where individual frees are optional and bulk deallocation happens on shutdown.

For graph replay and CUDA graph capture, the same caching allocator serves as the memory planner. During the first execution pass, the allocator handles allocation and reclamation normally while liveness tracking triggers frees at the right time. Since pool allocations produce stable pointers that persist across runs, the resulting memory layout can be frozen and replayed without invoking the allocator again. This avoids building a separate interference graph or running a register allocation algorithm. CUDA devices do not use this allocator at all; they rely on stream-ordered pool allocation managed entirely by the C++ backend.
